## 1. 项目概述

**项目名称**: DynamicWePubMonitor（微信公众号智能权重监控系统）

**一句话核心功能**: 针对体育/赛事类公众号的智能动态监控系统。根据公众号的历史更新频率、AI 分析的"体育赛事相关占比"和更新稳定性，实时动态调整抓取优先级（Tier 1~5）、选择抓取后端（WeRead 或 MP）和下次抓取间隔（24h~336h）。

**核心价值**:
- 解决传统公众号监控"盲目轮询、浪费资源、容易被封"的问题
- 让高价值体育赛事内容第一时间被发现并推送
- 支持团队多人协作管理上百个公众号
- 所有逻辑高度可配置、可视化、可审计

**架构风格**: 前后端分离（Vue 3 + FastAPI）

### 4.3 字段别名处理（重要）

后端模型使用 `current_tier` 和 `composite_score`，但前端期望使用 `tier` 和 `score`。

**解决方案**: 在 `AccountResponse` 中使用 `@computed_field` 添加别名：

```python
from pydantic import computed_field

class AccountResponse(BaseModel):
    id: int
    current_tier: int
    composite_score: float
    # ...

    @computed_field
    @property
    def tier(self) -> int:
        return self.current_tier

    @computed_field
    @property
    def score(self) -> float:
        return self.composite_score
```

**前端统一使用别名**（tier、score）。

## 6. 动态权重算法

### 6.1 评分公式

```
Score = 0.35*Frequency + 0.25*Recency + 0.25*Relevance + 0.15*Stability
```

### 6.2 各因子计算规则

#### Frequency（频率因子，权重 0.35）
- 近90天更新次数 + 密度（支持爆发型）
- 爆发型账号加分：单次更新 > 5 篇 + 15分

#### Recency（时效因子，权重 0.25）
- 上次更新距今时间 + 突发加成
- 突发加成：>30天未更新但本次≥3篇 → +35分

#### Relevance（相关度因子，权重 0.25）
- AI 返回的体育/赛事相关占比
- <50% 扣 15-20 分

#### Stability（稳定性因子，权重 0.15）
- 90天更新方差容忍
- 爆发+静默的高方差反而加分

### 6.3 Tier 分界阈值

| Tier | Score 范围 | 检查间隔 |
|------|-----------|---------|
| 1 | 80-100 | 24h |
| 2 | 65-80 | 48h |

| 4 | 35-50 | 144h |

### 6.4 边缘情况处理

- **长期静默后突然爆发**（类型A）：Recency 加分 +35
- **全年稳定高频**（类型B）：Stability 高分
- **季节性低频但特定时期高相关**（类型C）：Relevance 保护
- **AI 连续低相关**：连续3次 < 0.5 触发降级警告

## 9. 核心流程

### 9.1 扫码登录流程

```
1. 前端 POST /api/accounts/qr/generate {type: "weread"|"mp"}
2. 后端生成 ticket + mock 二维码数据，存储到 Redis（180s 过期）
3. 前端显示二维码，2s 轮询 GET /api/accounts/qr/status?ticket=xxx
4. 用户扫码 → 前端检测到 scanned → 用户确认 → confirmed
5. 后端保存 mock cookies → 创建 Account 记录 → 触发初始调度
6. 前端收到 confirmed → 关闭 Modal → 刷新列表
```

### 9.2 抓取闭环流程

```
1. APScheduler 触发 fetch_task.run_single_account(account_id)
2. scheduler_service 记录任务开始
3. fetcher_service:
   - 获取 account.current_tier
   - tier 1-2 → weread 后端
   - tier 3-5 → mp 后端
   - 从 proxy_service 获取对应服务类型代理
   - 调用 weread_fetcher 或 mp_fetcher 抓取新文章列表
4. parser_service:
   - 清洗 HTML → 纯文本 + 图片 URL
   - 下载图片到本地
5. ai_service:
   - 调用 LLM API（文字 + 图片 base64）
   - 返回 {ratio, reason, json_data}
6. article_service:
   - 保存所有 Article 记录
7. dynamic_weight_adjuster.update_after_fetch():
   - 计算新 score 和 tier
   - 更新 update_history, ai_relevance_history
   - 检查 manual_override 是否过期
8. scheduler_service:
   - 根据 next_interval 添加下一次 APScheduler 任务
9. notification_service:
   - 检查是否满足告警条件
10. log_service:
    - 记录完整操作日志
```

## 附录：经验总结

### 1. 数据库字段命名规范

**重要**: 前后端字段名必须一致，否则会导致 API 400 错误。

#### 后端 Account 模型
```python
current_tier      # 而非 tier
composite_score   # 而非 score
last_checked      # 而非 last_check
last_updated      # 而非 last_update
account_type      # 枚举值: weread, mp
status            # 枚举值: active, inactive, blocked
```

#### 解决方案
在 `app/schemas/account.py` 中使用 `computed_field` 添加别名：
```python
from pydantic import computed_field

class AccountResponse(BaseModel):
    id: int
    current_tier: int
    composite_score: float
    # ...

    @computed_field
    @property
    def tier(self) -> int:
        return self.current_tier
```

### 2. API 响应结构

后端返回的分页数据格式：
```json
{
  "total": 8,
  "items": [...],
  "page": 1,
  "page_size": 20,
  "total_pages": 1
}
```

前端必须使用 `response.data?.items` 而非 `response.data`。

### 3. 密码哈希问题

创建用户时必须使用 bcrypt 哈希密码：
```python
from passlib.context import CryptContext
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

hashed_password = pwd_context.hash(password)
user.hashed_password = hashed_password
```

验证密码：
```python
pwd_context.verify(plain_password, hashed_password)
```

### 4. SQLAlchemy 异步查询

使用 `scalars().all()` 或 `scalars().first()` 获取结果：
```python
result = await db.execute(select(User))
users = result.scalars().all()  # 返回列表
user = result.scalars().first()  # 返回单个或 None
```

### 5. SQLAlchemy 关系加载 (重要)

#### 问题背景
当 API 响应需要包含关联模型的字段时（如 Article 需要返回 Account 的 name），需要正确加载关系。

#### 解决方案
使用 `selectinload` 预加载关联对象：
```python
from sqlalchemy.orm import selectinload

# 在 Repository 中
result = await self.db.execute(
    Select(Article)
    .options(selectinload(Article.account))
    .where(...)
)
articles = result.scalars().all()
# article.account 已被加载
```

#### 常见错误
如果只返回 dict 而不通过 Pydantic model_validate，关联对象不会被序列化。

### 6. FastAPI 响应模型与额外字段

#### 问题
如果 Pydantic 响应模型的 `items` 类型定义为 `list[SomeModel]`，返回字典时额外的字段会被忽略。

#### 解决方案
创建专用的响应模型包含所有需要的字段：
```python
class ArticleWithAccountResponse(BaseModel):
    id: int
    account_id: int
    account_name: str | None = None
    # ... 其他字段
```
