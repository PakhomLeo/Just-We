# Just-We 参考项目详情

更新时间：2026-04-18

本文记录当前仓库内两个参考项目的定位、关键能力、代码入口和对本项目的映射方式。参考项目只作为行为和实现思路来源；当前主项目仍以 FastAPI、SQLAlchemy、Vue 3 的现行架构实现。

## 1. 本地参考项目

| 项目 | 本地路径 | 技术栈 | License | 当前用途 |
| --- | --- | --- | --- | --- |
| WeWe RSS | `external_references/wewe-rss` | NestJS、tRPC、Prisma、React/Vite | MIT | 微信读书登录、订阅源、文章列表、Feed、历史回填参考 |
| wechat-download-api | `external_references/wechat-download-api` | FastAPI、SQLite、httpx、curl_cffi | AGPL-3.0-only | 公众号后台扫码、文章列表、详情下载、内容处理、反风控参考 |

重要约束：

- WeWe RSS 是 MIT，可参考实现细节，但当前项目仍应按 Python/FastAPI 架构重写。
- wechat-download-api 是 AGPL-3.0-only，不能直接复制代码进入主项目；只能做 clean-room 行为参考，避免引入 AGPL 传染风险。

## 2. WeWe RSS

### 2.1 能力概览

WeWe RSS 的核心是“微信读书账号 + 公众号订阅源 + 自动刷新 + Feed 输出”。

关键能力：

- 微信读书平台二维码登录。
- 通过公众号分享链接解析公众号信息。
- 使用平台接口拉取公众号文章列表。
- 订阅源管理、历史文章回填、后台定时刷新。
- RSS、Atom、JSON Feed 输出。
- fulltext 模式按需下载原文并清理 HTML。
- OPML 导出。
- 今日小黑屋：账号 401/429 后当天避开。

| 能力 | 路径 | 说明 |
| --- | --- | --- |

### 2.3 对当前项目的映射

| WeWe RSS 概念 | 当前项目概念 | 当前实现 |
| --- | --- | --- |
| Account | `CollectorAccount(account_type=weread)` | `app/models/collector_account.py` |
| Feed | `MonitoredAccount` | `app/models/monitored_account.py` |
| Article | `Article` | `app/models/article.py` |
| 平台登录 | `WereadPlatformQRProvider` | `app/services/qr_providers.py` |

| 历史回填 | `FetchJobType.HISTORY_BACKFILL` | 目前是完整抓取包装，仍需补多页循环 |
| Feed 输出 | `FeedService` + `public_feeds.py` | 支持 RSS/Atom/JSON、单源/聚合 |

### 2.4 已吸收的实现思路

- `WEREAD_PLATFORM_URL` 默认指向兼容平台。
- WeRead 二维码返回 SVG data URL，前端可直接 `<img>` 展示。
- WeRead 平台文章接口路径兼容 `/api/v2/platform/mps/{id}/articles`。
- 监测对象拥有不可猜测 `feed_token`。
- Feed 输出支持 RSS、Atom、JSON。
- 支持 `mode=fulltext` 和标题 include/exclude。
- 支持 OPML 导出。
- 账号 401/429 应被纳入健康和风控状态，而不是反复请求。

### 2.5 仍建议继续吸收

- 历史回填需要真正按 page 递进，记录当前页并支持轮询进度。
- WeRead 账号小黑屋可以落到 `CollectorAccount.cool_until` 和 `last_error_category`。参考项目明确提示添加频率过高容易被封控、连续更新需要延迟，因此本项目避免用健康检查额外调用文章列表，并把单次疑似 401 先降级为冷却。
- Feed fulltext 可加 LRU 或持久缓存，避免阅读器重复触发原文下载。
- 全部刷新应按监测对象间增加延迟，减少风控概率。

## 3. wechat-download-api

### 3.1 能力概览

wechat-download-api 的核心是“公众号后台管理员扫码 + 后台接口获取任意公众号文章 + 详情下载 + 反风控 + RSS”。

关键能力：

- 微信公众平台扫码登录，保存 token、cookie、fakeid、nickname、过期时间。
- 调用 `searchbiz` 搜索公众号 fakeid。
- 调用 `appmsgpublish` 获取公众号文章列表。
- 使用 curl_cffi 模拟 Chrome TLS 指纹抓文章详情。
- 三层限频：全局、IP、文章间隔。
- SOCKS5/HTTP 代理轮换与失败冷却。
- 内容类型处理：普通文章、图文消息、短内容、音频、图片顺序。
- 微信 CDN 图片代理。
- 登录到期 24h/6h/过期通知。

| FastAPI 入口 | `external_references/wechat-download-api/app.py` | app、生命周期、路由、静态页面 |

### 3.3 对当前项目的映射

| wechat-download-api 概念 | 当前项目概念 | 当前实现 |
| --- | --- | --- |
| 公众号管理员扫码登录 | `CollectorAccount(account_type=mp_admin)` | `MpAdminQRProvider` |
| token/cookie 保存 | `CollectorAccount.credentials` | `CollectorAccountService.upsert_bound_account()` |
| fakeid 自动发现 | `discover_mp_admin_profile()` | `app/services/qr_providers.py` |
| `searchbiz` | `MonitoringSourceService.resolve_with_mp_admin_searchbiz()` | 创建监测对象补 fakeid |
| `appmsgpublish` | `MpAdminFetcher.fetch_updates()` | 列表抓取 |
| 文章详情下载 | `BaseChannelFetcher.fetch_article_detail()` | curl_cffi/httpx、风控识别 |
| 内容解析 | `ParserService`、`html_cleaner`、`image_downloader` | 解析正文、富文本、图片 |
| 代理池 | `Proxy`、`ProxyServiceBinding`、`ProxyService` | DB 代理池和服务绑定 |
| 限频 | `RateLimitService` | 内存/Redis 限频 |
| 通知 | `NotificationService` | 站内、邮件、Webhook |

### 3.4 已吸收的实现思路

- `mp_admin` 二维码图片转 data URL，避免微信图片热链问题。
- 登录链路包含 startlogin、scan 轮询、bizlogin 完成和 token 提取。
- 登录完成后尝试读取首页和 `searchbiz` 发现 fakeid。
- 详情下载使用浏览器请求头，优先 curl_cffi 模拟 Chrome TLS。
- 对登录页、验证码页、删除页、空页和 HTTP 错误做分类。
- 代理失败会降低成功率并设置冷却。
- 图片代理限制微信 CDN 域名。
- 凭证到期和健康异常进入通知系统。

### 3.5 仍建议继续吸收

- `mp_admin` 登录应增加浏览器 cookie 转发和更多微信响应兼容处理。
- 文章内容类型处理还应继续补真实样本，尤其是短内容、音频、视频、纯图片、多图文子项。
- 列表分页应在系统策略中明确每通道最大页数、每页数量、失败重试和停止条件。
- 限频统计应完整暴露到页面，按账号、代理、监测对象维度展示。
- Webhook 应加入失败重试和最近一次投递状态。

## 4. 当前项目吸收清单

已经基本对齐：

- WeRead 平台地址、二维码登录 provider、平台文章列表入口。
- mp_admin 登录 provider、二维码、scan、bizlogin、cookie/token/fakeid profile。
- 单源和聚合 RSS/Atom/JSON Feed。
- OPML/CSV 导出。
- 图片代理 `/api/image`。
- 代理服务绑定、代理冷却、成功率。
- 限频策略、随机延迟、风控分类。
- AI 三段 JSON pipeline。
- 站内通知、邮件、Webhook 配置。

仍需推进：

- 外部真实联调，不应只依赖本地测试替代。
- 历史回填完整分页进度。
- 代理策略真正读取系统配置中的最低成功率、直连禁用、粘性时长。
- 聚合 Feed token 前端可复制入口。
- 真实样本驱动 Parser 增强。

## 5. 联调建议顺序

1. 先验证 `mp_admin` 登录：生成二维码、扫码、确认、token/cookie/fakeid 落库。
2. 使用该账号搜索一个公开公众号，确认 `searchbiz` 能返回 fakeid。
3. 对该 fakeid 调用 `appmsgpublish`，确认列表分页和错误分类。
4. 对列表中的文章抓详情，确认正文、图片、发布时间、作者、封面。
5. 接入代理池并连续抓多篇，观察冷却、成功率、账号健康和通知。
6. 配置 AI API，确认三段 JSON 结果能入库。
7. 用 RSS 阅读器打开单源和聚合 Feed，确认 summary/fulltext 与图片代理。
