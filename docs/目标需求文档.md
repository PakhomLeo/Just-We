# DynamicWePubMonitor 目标需求文档

**文档目的**: 将项目的目标形态、核心需求、参考实现思路和交付边界整理清楚，作为后续产品设计、技术设计和开发排期的依据。

**最后更新**: 2026-04-16

## 1. 项目目标

本项目的目标不是做一个单一的 RSS 工具，而是做一个可登录、可管理、可调度、可扩展的微信公众号监测系统。

系统在用户登录后，允许其绑定两类可用抓取账号：

- 微信读书账号
- 微信公众号管理员账号

用户完成账号绑定后，可以在“公众号监测”页面添加目标公众号的文章链接。系统自动从链接中提取并补全公众号标识信息，例如：

- `biz`
- `fakeid`
- 公众号名称
- 账号来源信息

提取成功后，系统将目标公众号保存到数据库，并赋予默认监测等级、默认抓取策略、默认调度间隔。此后系统自动执行：

- 按等级选择不同抓取通道
- 获取公众号文章更新列表
- 抓取文章全文和图片
- 调用可配置 AI API 分析图文内容
- 保存分析结果
- 结合代理和反爬策略降低风控概率

## 2. 参考项目与吸收方向

本项目的实现方式参考以下三个项目，但目标是将其能力整合为统一平台，而不是简单复制。

### 2.1 wechat-download-api

参考地址：

- [tmwgsicp/wechat-download-api](https://github.com/tmwgsicp/wechat-download-api)

重点吸收能力：

- 公众号管理员微信扫码登录
- 使用公众号后台能力获取目标公众号文章列表
- 根据文章 URL 解析完整文章内容
- 反风控设计，包括代理池、限频、请求指纹模拟
- 面向 API 的服务化能力

### 2.2 wewe-rss

参考地址：

- [cooderl/wewe-rss](https://github.com/cooderl/wewe-rss)

重点吸收能力：

- 使用微信读书通道订阅公众号
- 通过分享链接添加公众号源
- 后台自动定时更新
- 账号状态管理，例如失效、封控、小黑屋
- 面向“监测源”的管理方式

### 2.3 we-mp-rss

参考地址：

- [rachelos/we-mp-rss](https://github.com/rachelos/we-mp-rss)

重点吸收能力：

- 更完整的后台管理界面
- 定时更新任务体系
- API / WebHook / AI Agent 接入思路
- 内容过滤和文章处理能力
- 前后端一体的管理体验

## 3. 核心业务目标

项目最终应实现以下闭环：

1. 用户登录系统
2. 用户在“抓取账号”页面扫码绑定自己的微信读书账号和微信公众号管理员账号
3. 用户在“公众号监测”页面粘贴目标公众号文章链接
4. 系统自动解析目标公众号信息并保存
5. 系统为该目标公众号分配默认等级和默认监控策略
6. 调度器根据等级自动选择抓取通道
7. 抓取到更新列表后继续抓取文章详情
8. 抓取完成后保存文章内容、图片、原始数据和结构化元数据
9. 调用可配置 AI API 对文章图文内容进行分析
10. 保存 AI 分析结果并作为后续分级、筛选、告警的输入

## 4. 用户角色与使用场景

### 4.1 用户角色

- `admin`：管理系统参数、账号、代理、AI 配置、监测规则
- `operator`：管理抓取账号、添加监测公众号、执行手动抓取
- `viewer`：查看结果、查看文章、查看分析结果

### 4.2 典型使用场景

- 运营人员绑定自己的微信读书账号和公众号管理员账号
- 运营人员提交某篇公众号文章链接，系统自动识别所属公众号
- 系统将该公众号纳入监测，并按等级持续轮询更新
- 系统发现新文章后抓取全文、图片和发布时间
- 系统调用外部 AI 服务分析是否为体育/赛事类内容、提取关键词、输出摘要
- 管理员根据结果调整监测等级、抓取频率和代理策略

## 5. 功能需求

### 5.1 认证与登录

系统需要支持平台用户登录，不等同于微信账号登录。

要求：

- 平台用户使用邮箱密码或既有后台账号登录
- 登录后基于 JWT / Session 维持状态
- 登录后才能访问抓取账号页、公众号监测页和监控配置页

### 5.2 抓取账号管理

抓取账号页是系统的基础能力页，需要支持两类账号绑定。

#### 5.2.1 微信读书账号绑定

要求：

- 用户进入抓取账号页后可以发起“微信读书扫码登录”
- 系统生成二维码并展示登录状态
- 用户扫码确认后，系统保存微信读书访问凭证
- 系统记录账号健康状态、最近可用时间、失效时间、风控状态

建议状态：

- 正常
- 失效
- 封控中
- 禁用

#### 5.2.2 微信公众号管理员账号绑定

要求：

- 用户可以发起“公众号管理员扫码登录”
- 系统生成微信公众号后台登录二维码
- 用户扫码确认后，系统保存后台访问凭证和 Cookie
- 系统自动记录 Cookie 有效期，并在到期前提醒

#### 5.2.3 账号与用户归属

要求：

- 抓取账号必须与平台用户关联
- 不同平台用户默认只能使用自己绑定的抓取账号
- 管理员可以查看所有抓取账号并决定是否允许共享调度

### 5.3 公众号监测源添加

用户在“公众号监测”页面添加目标公众号时，输入形式以文章链接为主。

要求：

- 输入任意合法微信公众号文章链接
- 系统自动解析链接中的 `biz`、文章 ID、可能的 mid/idx/sn 参数
- 系统进一步补全 `fakeid`、公众号名称、头像、来源通道
- 若数据库中不存在该公众号，则创建新监测记录
- 若已存在，则提示重复并支持继续沿用

创建监测记录时，系统默认写入：

- 默认等级，例如 Tier 3
- 默认抓取间隔
- 默认抓取策略
- 默认 AI 分析策略
- 默认代理策略

### 5.4 监测对象管理

每个被监测公众号应至少包含以下信息：

- 公众号名称
- `biz`
- `fakeid`
- 来源文章链接
- 当前等级
- 当前抓取通道
- 最近抓取时间
- 最近更新时间
- 当前状态
- 归属用户

状态建议：

- 监测中
- 暂停
- 封控观察
- 失效待修复

### 5.5 抓取策略与等级策略

系统需要根据公众号等级自动选择抓取方式，而不是固定走单一路径。

目标策略：

- 高等级公众号优先走更快、更稳定的通道
- 中低等级公众号走成本更低的通道
- 某个通道失效或封控时，允许降级或切换备用通道

建议分层：

- Tier 1-2：优先使用微信读书通道获取更新
- Tier 3-5：优先使用公众号管理员账号通道获取更新
- 当主通道失败时，允许基于规则尝试备用通道

说明：

- 以上是当前目标策略，不要求与现有代码完全一致
- 后续可继续扩展为“主通道 + 备用通道 + 熔断恢复”

### 5.6 更新列表抓取

系统应支持从不同通道获取目标公众号的更新列表。

#### 5.6.1 微信读书通道

目标能力：

- 通过微信读书相关接口获取公众号历史或最新文章列表
- 支持分页抓取
- 能区分新文章与已存在文章

#### 5.6.2 公众号管理员通道

目标能力：

- 通过公众号管理员后台已授权账号获取目标公众号文章列表
- 支持基于 `fakeid` 获取文章更新
- 支持定期轮询

#### 5.6.3 去重与增量

要求：

- 以 URL、文章主键或内容指纹作为去重依据
- 支持只抓新增文章
- 避免重复抓取相同全文内容

### 5.7 文章详情抓取与存储

当系统获得更新列表后，需要继续抓取文章详情。

要求：

- 抓取标题、作者、发布时间、封面、正文 HTML、纯文本正文、图片列表
- 下载或代理图片资源
- 存储原始响应和清洗后的结构化内容
- 保留文章和公众号之间的关联

建议存储层面区分：

- 原始文章数据
- 清洗后的可读内容
- AI 分析结果
- 调试用抓取元数据

### 5.8 AI 图文分析

系统需要提供可配置的 AI 分析能力，不绑定单一厂商。

要求：

- 支持用户或管理员配置 AI API 地址、Key、模型名
- 支持自定义 Prompt
- 支持对文章正文和图片进行联合分析
- 支持保存原始 AI 返回结果和结构化提炼结果

建议 AI 输出字段：

- 文章分类
- 体育/赛事相关度
- 摘要
- 关键词
- 风险标签
- 是否需要告警

### 5.9 反爬与动态 IP 管理

这是本项目的关键基础能力，不是附属功能。

要求：

- 系统内置代理池管理
- 不同抓取链路可配置不同代理策略
- 微信读书更新列表抓取可指定代理
- 微信读书文章详情抓取可指定代理
- 公众号后台更新列表抓取可指定代理
- 公众号文章详情抓取可指定代理
- AI 调用也可按需独立走代理

代理能力要求：

- 支持 HTTP / HTTPS / SOCKS5
- 支持按服务类型分组
- 支持成功率统计
- 支持自动轮换
- 支持失效摘除
- 支持手工禁用和手工测试

反爬策略要求：

- 请求频率控制
- 随机延迟
- User-Agent 和请求头伪装
- TLS 指纹或浏览器行为模拟
- 通道级熔断和恢复

### 5.10 调度与自动更新

系统不是一次性采集工具，而是持续监控系统。

要求：

- 每个公众号都有独立的下次抓取时间
- 调度器根据等级动态安排抓取频率
- 支持手动触发单个公众号抓取
- 支持手动触发全量更新
- 支持任务失败重试
- 支持失败统计和通道健康度统计

### 5.11 告警与可观测性

要求：

- 账号失效时产生告警
- Cookie 即将过期时产生告警
- 连续抓取失败时产生告警
- AI 判断命中高价值内容时可产生告警
- 系统保留操作日志、抓取日志和任务日志

## 6. 页面目标

### 6.1 抓取账号页

页面需要支持：

- 展示微信读书账号列表
- 展示公众号管理员账号列表
- 发起扫码登录
- 查看登录状态和有效期
- 查看账号健康状态
- 禁用或删除账号

### 6.2 公众号监测页

页面需要支持：

- 输入公众号文章链接添加监测对象
- 展示公众号名称、`biz`、`fakeid`、等级、状态、抓取通道
- 手动抓取
- 调整等级
- 暂停或恢复监测
- 查看最新文章和抓取历史

### 6.3 文章列表与详情页

页面需要支持：

- 查看文章基础信息
- 查看正文
- 查看图片
- 查看 AI 分析结果
- 查看原始抓取元数据

### 6.4 代理管理页

页面需要支持：

- 添加代理
- 按服务类型分组管理
- 查看成功率和最近检测结果
- 手动测试代理
- 禁用失效代理

### 6.5 系统设置页

页面需要支持：

- 配置 AI API
- 配置默认等级和默认抓取间隔
- 配置代理策略
- 配置反爬参数
- 配置告警渠道

## 7. 数据模型目标调整

为支撑上述需求，现有数据模型后续应至少补强以下几类实体或字段。

### 7.1 抓取账号实体

建议将“抓取账号”从当前 `Account` 语义中拆分或补充为独立概念，至少区分：

- `FetchCredential` 或 `CollectorAccount`
- 账号类型：`weread` / `mp_admin`
- 归属用户 ID
- 凭证内容
- 凭证过期时间
- 账号状态
- 最近成功时间
- 最近失败时间
- 风控状态

### 7.2 监测公众号实体

当前公众号监测对象应保留：

- `biz`
- `fakeid`
- `name`
- `tier`
- `score`
- `status`
- `primary_fetch_mode`
- `fallback_fetch_mode`

### 7.3 抓取任务实体

后续建议增加独立抓取任务记录，便于审计和重试：

- 任务类型
- 任务目标
- 通道类型
- 所用代理
- 开始时间
- 结束时间
- 执行结果
- 错误信息

## 8. 实现原则

后续实现应遵守以下原则：

- 平台用户登录与微信抓取账号登录必须分离
- 抓取账号与监测公众号必须分离
- 更新列表抓取与文章详情抓取必须分阶段处理
- 反爬、代理、限频必须是底层通用能力，不要散落在业务代码里
- AI 服务必须可替换、可配置，不与单一模型绑定
- 所有关键动作都应有状态、日志和可视化反馈

## 9. 当前阶段建议的开发顺序

建议按以下顺序推进，而不是同时铺开所有能力。

### 第一阶段：打通主流程

- 平台用户登录
- 抓取账号扫码绑定
- 公众号链接解析并入库
- 目标公众号列表展示
- 单通道抓取更新列表
- 单篇文章详情抓取
- 文章入库

### 第二阶段：双通道与调度

- 微信读书通道接入
- 公众号管理员通道接入
- 等级策略选择抓取通道
- 自动调度更新
- 去重和增量抓取

### 第三阶段：AI 与反爬

- 可配置 AI API
- 图文联合分析
- 代理池管理
- 通道级代理配置
- 限频与风控恢复

### 第四阶段：运营化能力

- 告警系统
- 日志中心
- 策略配置后台
- 通道健康度看板
- 手工重试与失败诊断

## 10. 非目标

当前目标文档不包含以下方向：

- 直接对外提供 RSS 作为唯一产品形态
- 做成纯阅读器产品
- 把所有参考项目的全部能力一次性照搬
- 先做复杂分布式集群，再做单机可用版本

当前更合理的目标是先做出“单机可部署、单用户可用、多抓取账号、多监测源、双通道抓取、AI 分析、可控代理”的完整闭环。

## 11. 一句话产品定义

DynamicWePubMonitor 的目标形态是：

**一个允许用户绑定自己的微信读书账号和微信公众号管理员账号、通过公众号文章链接自动建立监测对象、按等级动态选择抓取通道、结合代理和反爬策略持续抓取更新、并对图文内容执行可配置 AI 分析的微信公众号监测平台。**
