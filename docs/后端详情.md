# Just—We 后端详情

更新时间：2026-04-20

本文描述当前后端真实实现，包括模型、接口、服务、抓取流水线、配置、外部依赖和已知差距。当前后端主线是 FastAPI + SQLAlchemy Async + Alembic + Redis + APScheduler。

## 1. 技术栈

| 类型 | 当前实现 |
| --- | --- |
| Python | `>=3.12` |
| Web 框架 | FastAPI、Uvicorn |
| ORM | SQLAlchemy 2 async |
| 数据库 | PostgreSQL 默认，依赖 `asyncpg` |
| 迁移 | Alembic |
| 缓存/状态 | Redis，用于 QR 状态和限频窗口 |
| 调度 | APScheduler |
| HTTP | httpx，详情抓取可选 curl_cffi |
| HTML 解析 | BeautifulSoup、lxml |
| 图片处理 | Pillow、内部 image downloader |
| 测试 | pytest、pytest-asyncio |

启动入口：

- `app/main.py` 创建 FastAPI app。
- lifespan 内执行 `init_db()`、创建 media 目录、启动 scheduler、确保默认管理员、加载 active 监测对象排期、注册抓取账号健康检查任务。
- API 统一挂载在 `/api`，公开 Feed 挂载在根路径 `/feeds/...`。
- 媒体文件通过 `settings.media_url_prefix` 挂载，默认 `/media`。

## 2. 配置

配置文件：`app/core/config.py`。

主要环境变量：

- `DATABASE_URL`：默认 PostgreSQL async URL。
- `REDIS_URL`：默认 `redis://localhost:6379/0`。
- `JWT_SECRET_KEY`、`JWT_ALGORITHM`、`JWT_ACCESS_TOKEN_EXPIRE_MINUTES`。
- `LLM_API_URL`、`LLM_API_KEY`、`LLM_MODEL`。
- `WEREAD_PLATFORM_URL`、`WEREAD_PLATFORM_TIMEOUT_SECONDS`。
- `QR_CODE_EXPIRE_SECONDS`。
- `MEDIA_ROOT`、`MEDIA_URL_PREFIX`。
- `COLLECTOR_HEALTH_CHECK_HOUR` / `COLLECTOR_HEALTH_CHECK_MINUTE`：每天夜间自动健康检查时间。
- `ENSURE_DEFAULT_ADMIN`、`DEFAULT_ADMIN_ALIAS`、`DEFAULT_ADMIN_EMAIL`、`DEFAULT_ADMIN_PASSWORD`。

注意：

- `init_db()` 当前仍会 `Base.metadata.create_all`，适合开发启动兜底；生产应以 Alembic 为准。
- CORS 当前是 `allow_origins=["*"]`，生产需要改为显式配置。

### 3.6 Proxy

文件：`app/models/proxy.py`。

代理类型：

- `datacenter`。
- `isp_static`。
- `residential_static`。
- `residential_rotating`。
- `mobile_static`。
- `mobile_rotating`。
- `custom_gateway`。

轮换模式：

- `fixed`。
- `sticky`。
- `round_robin`。
- `per_request`。
- `provider_auto`。

服务键：

- `mp_admin_login`。
- `weread_login`。
- `mp_list`。
- `weread_list`。
- `mp_detail`。
- `weread_detail`。
- `image_proxy`。
- `ai`。

用途：

- 通过 `ProxyServiceBinding` 将代理绑定到登录、列表、详情、图片和 AI 等业务服务。
- 当前前端新增入口只暴露静态住宅和动态住宅两个 profile：静态住宅绑定登录和列表抓取，动态住宅绑定文章解析和图片下载；数据中心代理只允许绑定 AI。
- 抓取账号仍可绑定固定账号代理；未绑定时直连。
- 记录成功率、启停、冷却和最近错误。

### 3.7 系统配置

文件：`app/models/system_config.py`、`app/models/weight_config.py`。

配置实体：

- `AIAnalysisConfig`：AI URL、Key、模型、Prompt、文字/图片开关、目标文章类型、超时。
- `FetchPolicy`：Tier 阈值、检查间隔、Tier 抓取方式映射、重试、随机延迟、限频、代理、历史回填、通知策略。
- `NotificationEmailConfig`：SMTP 和 Webhook 配置。
- `WeightConfig`：权重比例、Tier 阈值、检查间隔、告警阈值。

## 4. API 路由

所有下列后台接口挂载在 `/api` 下。

- `GET /collector-accounts/`：当前可见抓取账号。

## 5. 核心服务

### 5.1 AuthService

职责：

- 密码校验。
- 默认管理员 alias 登录支持。
- JWT 创建和解析。
- 用户创建、最后登录时间更新。

### 5.2 QRLoginService 与 QR Providers

职责：

- Redis 存储二维码状态和 provider state。
- 生成 ticket、过期时间、二维码 URL。
- 轮询 provider 状态。
- confirmed 后创建或更新 `CollectorAccount`。

Provider：

- `WereadPlatformQRProvider`：对接兼容 WeWe RSS 的外部平台。
- `MpAdminQRProvider`：对接微信公众平台扫码链路。

### 5.3 CollectorAccountService

职责：

- 抓取账号创建、upsert、列表可见性。
- 绑定或解除账号代理，不清空旧凭证。
- 健康状态、失败分类、成功恢复。
- fakeid profile 更新。
- 判断账号是否可用于抓取。

### 5.4 MonitoringSourceService

职责：

- 解析文章 URL。
- 通过 WeRead 平台和 mp_admin searchbiz 补全公众号元数据。
- 创建监测对象。
- 按用户和 `biz` 去重。
- 初始化 Tier、分数、通道、下次抓取时间和能力标签。

### 5.5 FetcherService

职责：

- 为列表和详情选择代理。
- 根据账号类型分发到 `WeReadFetcher` 或 `MpAdminFetcher`。
- 提供统一 `fetch_updates()` 和 `fetch_article_detail()`。

`WeReadFetcher`：

- 尝试平台文章接口。
- 尝试公众号公开页 profile_ext。
- 解析 `msgList`、`general_msg_list` 和锚点链接。
- 无结果时回退到源文章 URL。

`MpAdminFetcher`：

- 要求 `fakeid` 和 cookies。
- 调用 `cgi-bin/appmsgpublish`。
- 解析 `app_msg_list` 或 `publish_page.publish_list[].appmsgex`。
- 支持多页配置。

详情抓取：

- 浏览器请求头。
- curl_cffi Chrome TLS 或 httpx。
- 代理候选。
- 微信验证码/风险页、登录页、删除页、空页识别。
- HTTP 状态和微信业务错误分类。

### 5.6 FetchPipelineService

职责：

- 完整抓取编排。
- 选择健康抓取账号。
- 创建 `FULL_SYNC`、`UPDATE_LIST`、`ARTICLE_DETAIL` 作业。
- URL 去重和已存在文章过滤。
- 详情抓取、解析、图片本地化。
- 将新文章标记为 `ai_analysis_status=pending` 并提交后台 AI 任务，不阻塞抓取成功返回。
- 保存文章。
- 高相关和连续低相关通知。
- 动态权重更新和下次调度。
- 抓取失败分类、账号状态更新、通知和重新排期。

当前历史回填：

- `run_history_backfill()` 目前是 `run_monitored_account()` 的包装。
- 会创建 `HISTORY_BACKFILL` 作业并记录 page/begin/fetched_count/saved_count。
- 还不是完整多页回填循环。

### 5.7 ArticleAIAnalysisService 与 AIService

职责：

- 按文章 ID 在独立数据库会话里执行后台 AI 分析。
- 读取系统 AI 配置。
- 文字分析。
- 图片分析。
- 目标类型判断。
- 严格 JSON 解析，失败重试一次。
- 输出 `success`、`failed`、`skipped` 状态。
- 选择 `ai` 服务绑定代理，代理失败时冷却并回退直连。

语义：

- `target_match` 只能是“是”或“不是”。
- “是”映射为 `ratio=1.0`，“不是”映射为 `ratio=0.0`。

### 5.8 ParserService

职责：

- 清洗微信公众号 HTML。
- 提取正文、富文本、内容类型、原始图片。
- 下载图片到本地 media。
- 输出可保存到 `Article` 的结构。

当前期望：

- 继续用真实样本增强短内容、音频、视频、纯图片、多图文等内容类型。

### 5.9 ProxyService 与 RateLimitService

ProxyService：

- 管理代理 CRUD。
- 校验代理和服务键兼容性。
- 选择代理。
- 成功提高成功率，失败降低成功率并设置冷却。
- 删除代理时只会清空引用该代理的抓取账号代理绑定，账号凭证保留并恢复直连。

RateLimitService：

- 支持全局、账号、代理、监测对象限频。
- 支持详情最小请求间隔。
- Redis 可用时用 Redis zset 记录窗口；不可用时回退进程内状态。

### 5.10 FeedService

职责：

- 生成单源 RSS/Atom/JSON Feed。
- 生成用户聚合 Feed。
- 支持 summary/fulltext。
- 支持标题 include/exclude。
- 导出 OPML/CSV。
- fulltext 模式替换原始图片 URL 为 `/api/image`。

### 5.11 NotificationService

职责：

- 站内通知列表、未读、已读、删除。
- 30 分钟内重复通知去重。
- 邮件通知投递。
- Webhook 通知投递。
- 高相关、连续低相关、抓取失败、代理池耗尽、抓取账号健康异常、凭证即将过期通知。

注意：

- 邮件和 Webhook 投递异常当前被吞掉，避免影响主流程；后续可记录投递失败状态。

## 6. 调度与任务

启动时：

- `start_scheduler()`。
- 加载 active monitored accounts 并按 `next_scheduled_at` 或间隔排期。
- 注册 `collector_health_checks` 周期任务。

任务文件：

- `app/tasks/fetch_task.py`：
  - `run_single_account()`。
  - `run_history_backfill()`。
  - `run_all_accounts()`。
- `app/tasks/health_task.py`：
  - `run_all_collector_health_checks()`。

## 9. 当前已知后端差距

优先级高：

- 外部真实联调未完成，不能把 WeRead、mp_admin、AI、SMTP 视为已生产验证。
- `FetchPipelineService` 当前按 Tier 抓取方式选择单一账号类型，不再执行 fallback 通道切换。
- 历史回填不是完整分页循环。
- 代理选择未显式过滤 `fail_until` 冷却代理，最低成功率固定在代码中。

优先级中：

- 文章列表筛选维度不足，AI 状态、目标命中、内容类型、标题关键词目前主要在前端过滤。
- 通知投递失败不记录状态。
- 邮件和 Webhook 仍缺少独立测试连接接口；AI 已有文字和图片测试连接接口。

优先级低：

- CORS 生产配置。
- `create_all` 与 Alembic 在生产启动边界上的明确切分。
- 更完整 OpenAPI 描述和错误码约定。

## 10. 运行命令

安装后端依赖：

```bash
uv sync
```

执行迁移：

```bash
uv run alembic upgrade head
```

启动后端：

```bash
uv run uvicorn app.main:app --reload
```

运行测试：

```bash
uv run pytest -q
```

默认本地依赖：

- PostgreSQL。
- Redis。
- 前端开发服务器会请求 `/api`，生产或跨端口开发时配置 `VITE_API_BASE_URL`。
