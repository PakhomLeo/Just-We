# Just-We 项目期望

更新时间：2026-04-18

本文描述项目希望达到的目标形态、功能边界和实现方式。它以当前代码为基础，不把尚需外部联调的能力写成已经生产可用。

## 1. 产品定位

Just-We 不是单纯 RSS 工具，而是面向微信公众号内容监测的前后端一体工作台。目标是让平台用户绑定自己的抓取账号，添加要监测的公众号，系统按策略持续拉取更新、抓取正文和图片、执行 AI 图文分析，并把账号、代理、限频、任务、通知和结果都放在一个可运维的后台里。

一句话目标：

让用户通过“抓取账号 + 监测对象 + 抓取流水线 + AI 分析 + Feed 输出 + 运维告警”的闭环，持续发现目标类型的公众号文章。

## 2. 当前代码基线

当前代码已经采用现行主模型：

- `CollectorAccount`：抓取账号，区分 `weread` 和 `mp_admin`。
- `MonitoredAccount`：被监测公众号，按平台用户和 `biz` 唯一。
- `Article`：文章正文、富文本、图片、AI 三段结果和抓取元数据。
- `FetchJob`：抓取流水线作业，包含 `full_sync`、`update_list`、`article_detail`、`history_backfill`。
- `Proxy` + `ProxyServiceBinding`：代理池与业务服务绑定。
- `AIAnalysisConfig`、`FetchPolicy`、`NotificationEmailConfig`、`WeightConfig`：系统配置和权重配置。

当前本地可验证的主能力：

- 平台用户登录、默认管理员引导、JWT 鉴权、角色入口控制。
- 抓取账号列表、二维码登录状态流、健康检查、fakeid 发现、账号代理绑定或解除。
- 从公众号文章链接创建监测对象，编辑名称、fakeid、状态和 Tier。
- 手动触发单个监测对象抓取、触发全部抓取、历史回填作业。
- 文章列表、文章详情、清洗富文本、图片、本地化路径、AI 三段结果、重跑 AI。
- 代理 CRUD、批量导入、服务绑定、连通性测试、失败冷却。
- 系统设置、权重配置、限频策略、邮件/Webhook 通知配置持久化。
- RSS/Atom/JSON 单源 Feed、聚合 Feed 后端接口、OPML/CSV 导出。

仍需外部联调证明的能力：

- WeRead 平台二维码登录和真实文章列表质量。
- 公众号后台扫码、Cookie/token 保活、fakeid 自动发现的真实稳定性。
- 微信文章列表接口和文章详情接口在真实风控环境下的成功率。
- 外部 AI API 的 JSON 稳定输出和图片多模态效果。
- SMTP/Webhook 在真实配置下的投递成功率。

## 3. 用户与权限期望

用户角色：

- `admin`：管理用户、系统设置、权重、代理、日志、全部可见资源。
- `operator`：管理抓取账号、监测对象、手动抓取、文章复核。
- `viewer`：查看仪表盘、文章和结果，不执行高风险写操作。

实现方式：

- 后端使用 `app/api/auth.py` 登录并返回 JWT。
- 前端 `frontend/src/stores/auth.js` 保存 token，并通过 `GET /api/auth/me` 拉取用户。
- 前端路由在 `frontend/src/router/index.js` 用 `requiresAuth`、`requiresAdmin` 控制页面入口。
- 后端管理类接口应继续使用 `AdminUser`、`CurrentUser`、资源 owner 检查，避免只依赖前端隐藏按钮。

期望补强：

- 后端角色判断应持续通过测试覆盖，确保 enum 和字符串角色比较不会导致权限判断偏差。
- 公开注册接口目前存在于后端 `POST /api/auth/register` 和前端 API 封装，但新版 UI 不提供公开注册入口；产品上应以管理员创建用户为准。

## 4. 抓取账号期望

抓取账号是系统访问外部微信/WeRead 链路的凭证载体，不等同于被监测公众号。

功能期望：

- 支持 `mp_admin` 公众号管理员扫码登录。
- 支持 `weread` 外部平台扫码登录。
- 保存凭证、外部标识、过期时间、健康状态、风险状态、最近成功/失败时间。
- `mp_admin` 和 `weread` 登录默认允许直连，用户可按账号手动绑定一个固定代理。
- 绑定或解除账号代理不清空旧凭证；后续健康检查负责发现凭证是否真实失效。
- 定期健康检查凭证，产生即将过期、已过期、风控、无效配置通知。

当前实现：

- `QRLoginService` 管理二维码 ticket、Redis 状态、provider 轮询和落库。
- `WereadPlatformQRProvider` 调用兼容 WeWe RSS 的平台登录接口。
- `MpAdminQRProvider` 直接对接 `mp.weixin.qq.com` 的 startlogin、二维码、scan、bizlogin 流程，并尝试发现 nickname/fakeid。
- `CollectorAccountService` 负责 upsert、健康标记、失败分类、成功恢复、fakeid profile 更新。
- `run_all_collector_health_checks` 按配置周期执行健康检查并通知。

期望补强：

- 为 `mp_admin` 登录链路增加更多真实响应样本测试，覆盖二维码生成失败、扫码确认失败、token 缺失、cookie 不完整、fakeid 未发现。
- 为 `weread` provider 明确平台响应契约，避免不同平台返回字段差异导致登录成功但凭证不可用。
- 账号列表应支持按 owner、类型、健康、风险和冷却状态分页筛选。

## 5. 监测对象期望

监测对象代表“某个平台用户希望持续监测的公众号”。

功能期望：

- 通过公众号文章 URL 创建监测对象。
- 从 URL 或外部解析结果提取 `biz`、`mid`、`idx`、`sn`，优先补全 `fakeid`、名称、头像、简介。
- 同一用户下同一个 `biz` 不重复创建，不同用户可以各自监测。
- fakeid 不可得时不伪造，明确标记为 `detail_only`。
- 保存 Tier、综合分、Tier 对应的唯一抓取方式、下次抓取时间、能力标签和策略配置。

当前实现：

- `MonitoringSourceService.parse_source_url()` 解析 URL 中的 `__biz` / `biz`，否则从路径尾部派生 `biz`。
- `resolve_with_weread_platform()` 尝试调用 WeRead 平台 `wxs2mp` 补全公众号元数据。
- `resolve_with_mp_admin_searchbiz()` 尝试使用健康 `mp_admin` 账号通过 `searchbiz` 搜索 fakeid。
- `MonitoredAccount` 使用 `(owner_user_id, biz)` 唯一约束。
- 默认 Tier 为 3，默认抓取方式由 `FetchPolicy.primary_modes` 决定；用户通过调整 Tier 间接切换抓取方式。

期望补强：

- URL 解析不应长期依赖路径尾部派生 `biz`，真实创建应优先要求能解析出可靠 `biz`。
- fakeid 搜索应支持更明确的查询来源，例如公众号名、alias、已登录账号搜索结果，而不是只依赖用户输入名称。
- 监测对象应继续增加按状态、Tier、能力、抓取方式、下次抓取时间的后端筛选。

## 6. 抓取流水线期望

抓取流水线应按阶段拆分，便于审计、重试和定位失败。

目标流程：

1. 调度或用户手动触发 `MonitoredAccount`。
2. 根据当前 Tier 对应的抓取方式选择健康 `CollectorAccount`。
3. 列表阶段使用账号代理或直连；详情阶段按服务绑定代理池轮换并在失败后直连兜底。
4. 拉取更新列表，写入 `UPDATE_LIST` 作业。
5. 按 URL 去重，只处理新文章。
6. 抓取文章详情，写入 `ARTICLE_DETAIL` 作业。
7. 解析 HTML，提取正文、清洗富文本、内容类型、原图和本地化图片。
8. 调用 AI 三段流水线。
9. 保存文章、更新权重、发送通知、写入 `FULL_SYNC` 结果。

当前实现：

- `FetchPipelineService.run_monitored_account()` 已按上述主流程编排。
- `FetcherService` 根据 `CollectorAccountType` 分发到 `WeReadFetcher` 或 `MpAdminFetcher`。
- `WeReadFetcher` 优先尝试平台文章接口，失败时尝试公众号公开页嵌入 JSON 和锚点回退。
- `MpAdminFetcher` 调用 `cgi-bin/appmsgpublish`，支持 `max_pages` 和 `page_size`。
- 详情抓取使用浏览器请求头、可选 `curl_cffi` Chrome TLS 指纹、代理候选、风控页面识别和错误分类。
- 失败分类目前包含 `configuration_error`、`credentials_invalid`、`risk_control`、`temporary_failure`，并写入 `FetchJob.payload`。

期望补强：

- 抓取方式切换应统一收敛到 Tier 配置，不在单个监测对象上暴露双通道差异。
- 历史回填目前是完整抓取包装，payload 记录 page/begin，但还不是完整多页进度循环；应按策略记录 page、begin、fetched_count、saved_count 并支持可中止循环。
- 更新列表和详情阶段应把实际 `proxy_id` 写入作业，而不仅是使用代理 URL。
- 抓取失败应继续细分永久不可用、验证码、登录失效、空页、临时网络、代理池耗尽。

## 7. AI 分析期望

AI 分析不绑定单一厂商，应由系统设置配置 URL、Key、模型和 Prompt。

目标能力：

- 文字全文解析：总结、关键词、实体、结构化信息。
- 图片内容解析：图片主题、文字识别、视觉信号。
- 目标类型判断：只输出 `target_match` 为“是”或“不是”。
- 合并结果保存到文章，后续参与权重和通知。

当前实现：

- `AIService` 以 `ArticleAIAnalysisService` 实现三段 pipeline。
- 请求格式兼容 OpenAI Chat Completions 风格，使用 `response_format: {"type": "json_object"}`。
- JSON 解析严格，允许去掉 markdown fence，失败后用格式错误提示重试一次。
- 图片最多取前 5 张转 data URL。
- AI disabled 时返回 skipped；失败时返回 failed 并写 `ai_analysis_error`。
- 权重中目标命中映射为“是”=1.0，“不是”=0.0。

期望补强：

- 为不同 AI provider 增加配置校验和测试连接。
- 保存每段原始模型响应摘要，便于追踪 JSON 失败。
- 图片路径目前必须是本地文件才会转 data URL；如果文章只保存公开 URL，需要在进入 AI 前确认是否已经本地化。

## 8. 代理与反风控期望

代理不是附属功能，而是微信链路稳定性的基础。

目标能力：

- 支持静态 ISP、静态住宅、静态移动、旋转住宅、旋转移动、自定义网关、数据中心代理。
- 登录和列表按抓取账号绑定一个固定代理，未绑定时直连。
- 详情、图片代理、AI 使用服务绑定代理池轮换。
- 失败代理冷却，成功率动态调整。
- 限频同时覆盖全局、账号、代理、监测对象和详情最小间隔。

当前实现：

- `ProxyService.compatible_service_keys()` 定义代理类型和业务服务兼容关系。
- `ProxyServiceBinding` 支持一个代理绑定多个服务键。
- `select_proxy()` 对详情、图片和 AI 等服务键选择代理；账号登录和列表走账号代理。
- `RateLimitService` 支持进程内状态，Redis 可用时用 Redis 记录限频窗口。
- 系统设置提供限频策略和代理策略持久化。

期望补强：

- `ProxyService._min_success_rate()` 当前固定 50%，应读取系统代理策略。
- 代理选择应过滤 `fail_until` 仍在冷却中的代理。
- 代理统计应按服务键输出健康度，便于前端展示“哪个链路没代理”。

## 9. Feed 与导出期望

Feed 是结果消费方式之一，不是系统唯一形态。

当前实现：

- 后端公开接口：`/feeds/{token}.rss|atom|json` 和 `/feeds/all/{token}.rss|atom|json`。
- 支持 `limit`、`page`、`mode=summary|fulltext`、`title_include`、`title_exclude`。
- `FeedService` 可导出 OPML 和 CSV。
- 单个监测对象有 `feed_token`，用户有 `aggregate_feed_token`。

期望补强：

- 前端监测页已经提供复制单源 Feed；聚合 Feed 复制当前依赖列表响应中的 token，但 `MonitoredAccountResponse` 不返回 `aggregate_feed_token`，应改为从当前用户 `authStore.user.aggregate_feed_token` 生成。
- RSS fulltext 模式可继续增加缓存，减少重复详情处理和图片代理压力。

## 10. 前端体验期望

前端应是可运维后台，而不是只展示数据的表格。

当前实现：

- v2 页面位于 `frontend/src/views/v2/`，统一由 `V2AppShell` 承载。
- URL 保持：`/dashboard`、`/capture-accounts`、`/mp-accounts`、`/articles`、`/articles/:id`、`/proxies`、`/logs`、`/system-users`、`/weight`、`/settings`。
- 顶部胶囊导航、通知抽屉、主行动按钮、页面 action rail 已统一。
- 业务页面基本覆盖当前后端接口。

期望补强：

- 修复前端枚举值与后端枚举不一致的表单项，例如代理 `rotation_mode` 后端期望 `fixed`、`sticky`、`round_robin`、`per_request`、`provider_auto`，当前页面部分使用 `static`、`rotating`。
- 去除页面文案中“功能说明式”的冗长描述，保留必要状态和风险提示。
- 增加前端自动化 smoke 测试，覆盖登录、路由守卫、核心 API 页面加载、表单保存。

## 11. 验收边界

本地验收：

- `pytest -q` 通过。
- `cd frontend && npm run build` 通过。
- 本地 PostgreSQL 和 Redis 可用。
- 登录、列表、配置保存、任务入库、文章详情和通知抽屉可用。

外部联调验收：

- WeRead 二维码从生成到确认，再到凭证落库和文章列表拉取成功。
- mp_admin 二维码从生成到确认，再到 token/cookie/fakeid 落库成功。
- 至少一个真实公众号可拉列表、抓详情、解析正文和图片。
- AI API 能返回合法 JSON，三段结果入库。
- SMTP/Webhook 能收到真实告警。

非目标：

- 不做纯阅读器产品。
- 不把所有参考项目代码直接复制进主仓库。
- 不先做复杂分布式调度，再做单机闭环。
- 不把 fakeid 不可得的监测对象伪造成列表抓取可用。
